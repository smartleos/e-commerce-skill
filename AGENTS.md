# AGENTS.md

本文件是本项目的协作说明，适用于在本仓库内工作的 Codex / Agent。

## 回复语言

- 默认全程使用中文回复用户。
- 除非用户明确要求英文，否则不要把交付内容写成英文。
- 可以保留必要的英文专有名词、命令、文件名、字段名和代码标识。

## 项目定位

`e-commerce-skills` 是一个电商技能市场包，用来沉淀可复用的电商工作流能力，例如商品策略、详情页策划、卖点提炼、直播带货脚本、货架内容优化、活动复盘和平台运营分析。

项目中的每个 skill 都应该聚焦一个明确电商任务，并输出可直接执行、可复用、可扩展的工作流说明、脚本或结构化结果。

## Skill 编写规则

- 创建新 skill 时，不要急着直接写文件。先向用户提出一组关键问题，帮助梳理场景、目标用户、输入、输出、边界、触发条件和质量标准。
- 问题澄清后，先和用户做一轮头脑风暴，整理可能的 workflow、使用场景、输出形态和差异化价值。
- 头脑风暴完成后，先给出拟定步骤和 skill 结构，等待用户确认。
- 只有在用户确认方向和步骤后，才正式写入或修改 `skills/<skill-name>/SKILL.md`。
- 新增或修改 `skills/<skill-name>/SKILL.md` 时，正文内容必须使用中文。
- frontmatter 字段名保持工具要求的英文键名，例如 `name`、`description`。
- frontmatter 的字段值优先使用中文；只有名称、行业术语或兼容性要求需要英文时才保留英文。
- skill 内容要面向实际执行，不写泛泛的电商理论。
- 每个 skill 应尽量包含：
  - 使用场景
  - 触发条件
  - 必要输入
  - 工作流程
  - 输出格式
  - 质量标准
  - 不足信息时的默认处理方式
- 缺少非关键字段时，优先做轻量合理假设，并在输出中标注“假设”。
- 缺少关键字段且无法合理推断时，再向用户追问。
- 不编造平台政策、GMV、转化率、行业 benchmark、投放效果、客户案例或未核验的竞品数据。
- 如果需要最新平台规则、公开榜单、价格、店铺数据、投放机制或电商合规要求，先联网核验，并在输出中标注来源和时间。

## 目录约定

- `skills/<skill-name>/SKILL.md`：单个 skill 的主说明文件。
- `skills/<skill-name>/scripts/`：该 skill 可复用的脚本。
- `commands/`：命令入口说明。
- `.codex-plugin/plugin.json`：Codex 插件清单。
- `.claude-plugin/`：兼容 marketplace 风格的插件描述。
- `README.md` 与 `README.zh.md`：项目说明；当新增 skill 或改变项目定位时同步更新。

## 开发约定

- 优先沿用项目已有结构，不为单个小功能引入复杂框架。
- 修改范围要贴近用户请求，避免顺手重构无关文件。
- 读文件、查找内容优先使用 `rg` / `rg --files`。
- 手动编辑文件优先使用 `apply_patch`。
- 新增脚本时默认保持简单、可直接运行，并在 README 或 skill 中给出调用方式。
- 如果脚本会生成文件，说明输出路径和覆盖策略。
- 提交前尽量运行与改动相关的最小验证命令。

## Git 与推送约定

- 用户说“推送”的时候，默认含义是：
  1. 检查当前改动；
  2. 只暂存与本次任务相关的文件；
  3. 创建清晰的提交；
  4. 推送到 GitHub 远端。
- 在执行提交或推送前，先确认是否存在明显无关或用户未说明的工作区改动。
- 不要回滚用户已有改动，除非用户明确要求。
- 不要把无关文件混入本次提交。

## 输出质量

- 输出要能直接给商品、运营、内容、直播、投放、设计、客服或销售团队使用。
- 结论要回到商品价值、人群需求、平台场景、转化路径、货架内容或数据动作。
- 表格可以用于结构化信息，但不要为了显得完整而堆砌空泛字段。
- 对不确定内容要明确标注，不用虚构数据填补空白。
