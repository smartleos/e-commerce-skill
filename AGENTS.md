# AGENTS.md

本文件是本專案的協作說明，適用於在本儲存庫內工作的 Codex / Agent。

## 回覆語言

- 預設全程使用中文回覆使用者。
- 除非使用者明確要求英文，否則不要把交付內容寫成英文。
- 可以保留必要的英文專有名詞、命令、檔名、欄位名和程式碼標識。

## 專案定位

`e-commerce-skills` 是一個電商技能市場包，用來沉澱可複用的電商工作流能力，例如商品策略、詳情頁策劃、賣點提煉、直播帶貨腳本、貨架內容最佳化、活動復盤和平台營運分析。

專案中的每個 skill 都應該聚焦一個明確電商任務，並輸出可直接執行、可複用、可擴展的工作流說明、腳本或結構化結果。

## Skill 編寫規則

- 建立新 skill 時，不要急著直接寫檔案。先向使用者提出一組關鍵問題，幫助梳理場景、目標使用者、輸入、輸出、邊界、觸發條件和品質標準。
- 問題澄清後，先和使用者做一輪腦力激盪，整理可能的 workflow、使用場景、輸出形態和差異化價值。
- 腦力激盪完成後，先給出擬定步驟和 skill 結構，等待使用者確認。
- 只有在使用者確認方向和步驟後，才正式寫入或修改 `skills/<skill-name>/SKILL.md`。
- 新增或修改 `skills/<skill-name>/SKILL.md` 時，正文內容必須使用中文。
- frontmatter 欄位名保持工具要求的英文鍵名，例如 `name`、`description`。
- frontmatter 的欄位值優先使用中文；只有名稱、行業術語或相容性要求需要英文時才保留英文。
- skill 內容要面向實際執行，不寫泛泛的電商理論。
- 每個 skill 應儘量包含：
  - 使用場景
  - 觸發條件
  - 必要輸入
  - 工作流程
  - 輸出格式
  - 品質標準
  - 不足資訊時的預設處理方式
- 缺少非關鍵欄位時，優先做輕量合理假設，並在輸出中標註“假設”。
- 缺少關鍵欄位且無法合理推斷時，再向使用者追問。
- 不編造平台政策、GMV、轉換率、行業 benchmark、投放效果、客戶案例或未核驗的競品資料。
- 如果需要最新平台規則、公開榜單、價格、店鋪資料、投放機制或電商合規要求，先連網核驗，並在輸出中標註來源和時間。

## 目錄約定

- `skills/<skill-name>/SKILL.md`：單個 skill 的主說明文件。
- `skills/<skill-name>/scripts/`：該 skill 可複用的腳本。
- `commands/`：命令入口說明。
- `.codex-plugin/plugin.json`：Codex 外掛清單。
- `.claude-plugin/`：相容 marketplace 風格的外掛描述。
- `README.md` 與 `README.zh.md`：專案說明；當新增 skill 或改變專案定位時同步更新。

## 開發約定

- 優先沿用專案已有結構，不為單個小功能引入複雜框架。
- 修改範圍要貼近使用者請求，避免順手重構無關檔案。
- 讀檔案、查詢內容優先使用 `rg` / `rg --files`。
- 手動編輯檔案優先使用 `apply_patch`。
- 新增腳本時預設保持簡單、可直接執行，並在 README 或 skill 中給出呼叫方式。
- 如果腳本會生成檔案，說明輸出路徑和覆蓋策略。
- 提交前儘量執行與改動相關的最小驗證命令。

## Git 與推送約定

- 使用者說“推送”的時候，預設含義是：
  1. 檢查當前改動；
  2. 只暫存與本次任務相關的檔案；
  3. 建立清晰的提交；
  4. 推送到 GitHub 遠端。
- 在執行提交或推送前，先確認是否存在明顯無關或使用者未說明的工作區改動。
- 不要回滾使用者已有改動，除非使用者明確要求。
- 不要把無關檔案混入本次提交。

## 輸出品質

- 輸出要能直接給商品、營運、內容、直播、投放、設計、客服或銷售團隊使用。
- 結論要回到商品價值、人群需求、平台場景、轉換路徑、貨架內容或資料動作。
- 表格可以用於結構化資訊，但不要為了顯得完整而堆砌空泛欄位。
- 對不確定內容要明確標註，不用虛構資料填補空白。
