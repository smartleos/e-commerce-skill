# 電商詳情頁內容黑白線框稿

## 觸發條件

當使用者希望梳理電商商品詳情頁內容時，使用本 skill。

只要使用者的需求涉及以下任一方向，就應觸發：

- 想為某個商品整理詳情頁內容結構
- 想把產品資訊轉成詳情頁文案
- 想生成手機長圖形式的詳情頁 HTML
- 想看到一版低保真黑白線框稿效果
- 想圍繞核心價值、使用者痛點、產品賣點、使用場景、購買理由來組織詳情頁
- 想把商品資訊整理成可交給營運、設計、內容團隊繼續執行的詳情頁策劃稿

## 核心原則

- 先收集精準資訊，再生成詳情頁內容。
- 不使用“隨便輸入也先出初稿”的邏輯。
- 使用者必須先提供完整產品資訊表，再進入內容生成。
- 如果資訊不完整，先讓使用者補齊關鍵欄位。
- 按固定步驟推進，不讓使用者選擇先精修哪一層。
- 每一步都給多個選擇，並告訴使用者如何選擇、融合、修改或補充資訊。
- 使用者確認或選擇後，再進入下一步。
- 所有確認內容最終先整理成一個完整的詳情頁內容文件。
- 內容文件確認後，再生成一份完整可執行的 HTML 黑白線框稿。
- 詳情頁的目的必須回到銷售轉換；每一屏都要有明確的使用者感受、購買理由或行動推進作用。
- 規劃過程必須區分「內部策劃欄位」與「可上畫面文案」。內部分析、選擇理由、資訊任務、設計指示、風險提醒和假設說明不得出現在 HTML 或後續設計成品中。
- CTA 不得只放最後一屏。預設在首屏、中段節點與末屏分段配置，讓使用者在不同閱讀深度都能採取行動。

## 觸發後的開始語

觸發後不要直接生成內容，先使用以下固定開始語收集產品資訊：

```text
我會先基於完整產品資訊，幫你一步步梳理電商詳情頁內容；確認內容後，再整理成詳情頁內容文件，並生成一份可執行的手機長圖 HTML 黑白線框稿。

這份黑白線框稿用於快速預覽詳情頁內容結構、模組順序和轉換邏輯，方便你和營運、設計、內容團隊對齊方向後再進入視覺設計。

請先補充產品資訊，包括但不限於：

【產品名】：[填寫你的產品名]
【產品定位】：[填寫產品定位]
【目標人群】：[填寫目標人群]
【核心場景】：[填寫目標場景]
【核心賣點】：[列出所有核心賣點]

如果你暫時不確定某一項，可以先寫“待定”，但產品名、目標人群和核心賣點建議儘量具體，這會直接影響詳情頁內容的準確度。
```

## 必要輸入

開始前，先要求使用者填寫以下產品資訊表：

```text
產品資訊如下：
【產品名】：[填寫你的產品名]
【產品定位】：[填寫產品定位]
【目標人群】：[填寫目標人群]
【核心場景】：[填寫目標場景]
【核心賣點】：[列出所有核心賣點]
```

如果使用者沒有提供完整資訊，不要直接生成 HTML。先提示使用者補齊：

```text
請先補充完整產品資訊，我會基於這些資訊一步步生成詳情頁內容。

產品資訊如下：
【產品名】：[填寫你的產品名]
【產品定位】：[填寫產品定位]
【目標人群】：[填寫目標人群]
【核心場景】：[填寫目標場景]
【核心賣點】：[列出所有核心賣點]
```

## 工作流程

收到完整產品資訊後，按以下順序推進：

1. 核心價值承諾層
2. 痛點錨定層
3. 賣點強化層
4. 使用場景層
5. 產品基礎資訊補充
6. 決策收口層
7. 逐屏編排與 CTA 佈局
8. 整理詳情頁內容大文件
9. 生成完整 HTML 黑白線框稿

不要跳步。不要讓使用者選擇先做哪一層。

每一步的固定節奏是：

1. 基於使用者提供的產品資訊和上一輪確認內容進行分析。
2. 給多個可選方案。
3. 用清晰話術指引使用者選擇、融合、修改或補充資訊。
4. 使用者確認後，記錄該步驟結果。
5. 自動進入下一步。

## 1. 核心價值承諾層

目標：將產品的所有賣點提煉成一句有差異點和共鳴感的核心價值承諾，適合放在電商詳情頁第一屏。

要求：

- 用詞精練有力
- 朗朗上口
- 能激發目標人群共鳴
- 適合做第一屏關鍵定義
- 不虛構產品沒有提供的資訊

輸出 10 句話備選。

輸出格式：

```text
核心價值承諾備選：
1.
2.
...
10.

你可以這樣回覆我：
- 直接選一個編號，例如：選 3
- 選多個讓我融合，例如：3 和 8 結合
- 說明你想要的方向，例如：更高階、更直接、更有情緒、更專業、更像天貓首屏
- 如果都不滿意，告訴我“不滿意 + 原因”，我會重新生成一組
```

## 2. 痛點錨定層

基於使用者選中的核心價值承諾，挖掘使用者的痛點、癢點、爽點。

分析維度：

- 痛點：使用者在此情境中最擔憂或最想解決的問題是什麼？
- 癢點：使用者希望呈現出的理想狀態、身份感或角色投射是什麼？
- 爽點：使用者在哪一瞬間獲得情緒滿足、輕鬆感或被理解的感受？

然後提煉出適合電商詳情頁的一句話表達，給 10 個選擇。

輸出格式：

```text
痛點：

癢點：

爽點：

詳情頁表達備選：
1.
2.
...
10.

你可以這樣回覆我：
- 直接選擇一句詳情頁表達，例如：選 5
- 讓我融合多個方向，例如：2 和 9 結合
- 告訴我痛點要更尖銳、癢點要更有嚮往感，或爽點要更有被理解的感覺
- 如果你覺得痛點不準，可以補充真實使用者抱怨、差評、諮詢問題或購買前顧慮
```

## 3. 賣點強化層

目標：精準翻譯產品賣點，將功能轉換成使用者真正關心的購買理由。

對每一個賣點按以下步驟處理：

1. 提煉賣點：識別最重要的功能、成分、工藝、標準或差異化特徵。
2. 轉換買點視角：說明它能解決什麼問題、帶來什麼好處。
3. 文案再造：寫成一句更有感知力、更有畫面感的詳情頁買點文案，每句控制在 25 字以內。

輸出格式：

```text
賣點 1
主標題：使用者買點
副標題：產品賣點

賣點 2
主標題：使用者買點
副標題：產品賣點

你可以這樣回覆我：
- 確認保留，例如：這組可以
- 指定某個賣點重寫，例如：賣點 2 重寫
- 告訴我某個賣點要更理性、更生活化、更有畫面感或更適合詳情頁大標題
- 如果賣點有證據，例如成分、工藝、認證、資料、材質、使用方法，可以繼續補充，我會把它轉成更有說服力的買點
```

如果使用者給的賣點沒有證據，不要補造資料。可以用體感、場景或結果表達，但避免虛假承諾。

文案品質要求：

- 優先使用痛點、癢點、爽點、具體情境、動作、結果感或共感來表達。
- 避免只平鋪直述產品資訊、空泛形容、無情緒、無畫面、無動作或無法讓目標人群代入的文案。
- 成分、規格、使用方法、適用邊界及必要告知可以使用客觀說明文字，但仍要清楚易讀。

## 4. 使用場景層

目標：把產品從抽象賣點落到真實購買和使用情境。

按以下步驟完成：

1. 基於產品資訊和核心賣點，系統拆解可能出現的使用場景。
2. 根據市場大小、品牌優勢和轉換價值，篩選 Top 4 高價值場景。
3. 對 Top 4 高價值場景做極簡概括。

輸出格式：

```text
全量使用場景：
- 場景：
  價值：

Top 4 高價值場景：
1. 場景名：一句話概括
2. 場景名：一句話概括
3. 場景名：一句話概括
4. 場景名：一句話概括

你可以這樣回覆我：
- 確認保留這 4 個場景
- 替換某個場景，例如：把第 3 個換成通勤場景
- 告訴我哪個場景最重要，我會把它放到詳情頁更靠前的位置
- 如果你有真實銷售場景、使用者反饋、直播話術或小紅書種草場景，也可以補充，我會重新篩選 Top 4
```

## 5. 產品基礎資訊補充

目標：讓使用者補充產品基礎參數、成分、規格、用法、適用邊界、認證、研發資料等事實性資訊，用於後續整理內容文件和 HTML 線框稿。

這一部分以使用者輸入為準，不由 AI 編造。尤其是兒童、食品、健康相關產品，凡是涉及參數、成分、功效、認證資訊等事實性內容，都必須要求使用者提供或後續自行核查。

可以引導使用者補充：

- 產品規格
- 關鍵成分或配方
- 使用方法
- 適用人群與不適用人群
- 功效、標準、檢測、認證或研發資料
- 包裝、口味、容量、材質等基礎參數
- 需要謹慎表達或不能承諾的邊界

輸出格式：

```text
產品基礎資訊補充：
規格參數：
成分/配方：
使用方法：
適用邊界：
認證/檢測/研發資料：
其他需要進入詳情頁的資訊：

你可以這樣回覆我：
- 直接補充具體資訊，例如：規格是 250ml，適合 3 歲以上兒童
- 某一項沒有就寫“暫無”或“待核查”
- 如果有檢測報告、研發資料、認證資訊，可以直接貼出來，我只基於你提供的內容整理
- 如果涉及兒童、食品、健康相關表述，請優先提供真實研發資料和可核查資訊
```

## 6. 決策收口層

目標：在詳情頁底部完成購買理由總結，降低使用者猶豫。

可以從以下方向收口：

- 安心感
- 專業感
- 場景必要性
- 價效比
- 禮贈價值
- 低試錯成本
- 長期使用價值

輸出 3 到 5 組收口方案。

輸出格式：

```text
決策收口備選：
1. 主標題：
   副標題：
   行動提示：

2. 主標題：
   副標題：
   行動提示：

你可以這樣回覆我：
- 直接選擇一組，例如：選 2
- 讓我融合兩組，例如：1 和 4 結合
- 告訴我收口方向，例如：更強轉換、更剋制、更高階、更有安全感、更突出價效比
- 如果使用者下單前還有具體顧慮，例如價格、規格、適用人群、效果擔心、售後問題，可以補充，我會把它寫進收口層
```

## 7. 逐屏編排與 CTA 佈局

目標：把已確認內容編排成可直接製作的逐屏詳情頁，並在不同閱讀深度設置自然、明確的行動入口。

CTA 配置規則：

- 預設採用「首屏、每隔約 3 屏、末屏」的節奏。
- 10 屏詳情頁預設在第 1、4、7、10 屏配置 CTA；最後一屏即末屏 CTA。
- 非 10 屏詳情頁依總屏數彈性調整。4 屏以上原則是首屏必有、中段至少一處、末屏必有；1 至 3 屏可合併首屏與末屏 CTA，不強制另設中段 CTA。
- 避免連續多屏堆疊 CTA；短頁也不要為了湊數讓每一屏都出現相同 CTA。
- 第 1 屏 CTA 承接核心價值承諾；中段 CTA 應放在重要賣點、證據或高價值場景之後；末屏 CTA 承接決策收口。
- CTA 可以是購買、選規格、領券、加入購物車、立即體驗、了解方案等，但只能使用使用者已確認且頁面實際支援的行動。
- 不虛構價格、折扣、贈品、庫存、限時、銷量、售後承諾或平台功能。
- 每個 CTA 都要和當前屏內容形成因果承接，不使用與上下文無關的制式按鈕文案。

每一屏必須分成兩類資訊：

```text
第 X 屏

【內部策劃欄位｜不得進入成品】
資訊任務：
內容選擇理由：
建議視覺方向：
風險/待核查：

【可上畫面文案｜僅此區可進入 HTML 與視覺設計】
主標題：
副標題：
輔助文案：
CTA：（非 CTA 屏填“無”）
必要告知文字：（沒有則填“無”）
```

逐屏編排完成後必須自檢：

- 每屏主文案是否聚焦單一重點，沒有把規劃討論、分析結論或製作指示寫進消費者文案。
- 除成分、規格、用法和必要告知外，每屏是否至少包含痛點、癢點、爽點、情境共感、使用動作、結果感、購買利益或行動推進中的一項。
- 文案是否避免平鋪直述、空泛、無聊、無情緒、無動作、無法同理或沒有共感。
- CTA 是否出現在合理節點，且文案與當前屏內容、實際購買路徑一致。
- 所有事實、資料、價格、優惠、認證和承諾是否有使用者提供的依據。

如果某屏文案未通過自檢，先重寫該屏再交付，不要把不合格文案留給視覺設計階段處理。

## 最終交付：內容文件與 HTML

當前六層內容都確認並完成逐屏編排後，必須先整理完整的詳情頁內容文件。

內容文件包括：

- 產品資訊
- 核心價值承諾
- 痛點、癢點、爽點
- 詳情頁表達句
- 每個賣點的產品賣點、使用者買點、主標題、副標題
- Top 4 使用場景
- 產品基礎資訊補充
- 決策收口方案
- 逐屏內容順序
- 第 1 屏、中段節點與末屏的 CTA 配置
- 每屏的「內部策劃欄位」與「可上畫面文案」分區

內容文件輸出後，用以下話術引導使用者確認：

```text
你可以這樣回覆我：
- 回覆“確認”，我會繼續生成完整 HTML
- 指定修改某一層，例如：核心價值承諾層再改一下
- 告訴我 HTML 呈現偏好，例如更像天貓詳情頁、更像小紅書種草長圖、更像品牌官網低保真稿
- 如果不需要 HTML，也可以讓我只保留內容文件
```

內容文件確認後，再生成完整可執行的 HTML 黑白線框稿。

HTML 要求：

- 手機長圖詳情頁形式，豎向滾動
- 主容器寬度控制在 `375px` 左右並居中展示
- 白底、黑字、淺灰模組
- 低保真黑白線框稿風格
- 每一屏之間用細黑線分隔
- 標題大、粗、短句化
- 用灰色卡片、圓形節點、簡單線條圖、流程圖表達內容
- 不依賴外部圖片、字型、CDN 或第三方庫
- CSS 寫在同一個 HTML 檔案內
- HTML 畫面只能使用各屏「可上畫面文案」區的文字。
- 不得把資訊任務、內容選擇理由、設計指示、風險提醒、假設、對話話術、自檢結果或其他內部溝通文字寫入 HTML。
- 按逐屏編排保留 CTA，10 屏預設在第 1、4、7、10 屏呈現；非 10 屏按已確認的彈性節點呈現。
- CTA 應是清晰可辨識的行動區域，文案必須與當前屏內容一致。

輸出 HTML 前，用一句話說明：

```text
本框架圖僅作為內部探討示例，AI生成的內容都可能有問題。尤其是兒童、食品、健康相關產品，凡是涉及參數、成分、功效、認證資訊等這類事實性內容，應該以你的研發資料為準，做好事實核查。

下面是根據已確認內容生成的完整 HTML，可直接儲存為 .html 檔案執行。
```

以上提醒只放在程式碼區塊外，不得成為 HTML 頁面或任何一屏的可見文案。

## 預設處理方式

- 缺少產品資訊表：先讓使用者補齊，不直接生成。
- 缺少關鍵欄位：先追問，不跳到內容生成。
- 缺少非關鍵細節：可以做輕量假設，但必須標註“假設”。
- 使用者中途要求重寫某一層：只重寫該層，並說明最終內容文件和 HTML 會同步更新。
- 使用者要求直接生成 HTML：如果產品資訊完整，可以直接生成；如果資訊不完整，先補齊關鍵欄位。

## 品質標準

- 內容必須基於使用者提供的產品資訊。
- 文案必須回到商品價值、人群需求、平台場景和轉換路徑。
- 不寫泛泛的電商理論。
- 不堆空泛欄位。
- 不編造未經核驗的資料、政策、案例、認證或競品事實。
- 頁面結構要清楚，模組要像可交給設計師繼續細化的低保真線框稿。
- CTA 必須分布在首屏、中段與末屏；10 屏預設檢查第 1、4、7、10 屏。
- 成品可見文字只能來自「可上畫面文案」，不得混入內部策劃、分析、設計或自檢文字。
- 除成分說明與必要告知外，文案要有痛點、癢點、爽點、情境、共感、動作、利益或轉換推進，不能只做無情緒的資訊陳列。
